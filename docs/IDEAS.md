## 9. 画像入力 / 認識系

スクリーンショット・写メから盤面を取り込めると、「手元のゲームの局面をその場で AI に問う」「誰かが SNS に上げた局面を試す」が一気に可能になる。
編集モードの上位互換。手で塗らずに済むので体験のジャンプが大きい。

| | 項目 | 見積 | メモ |
|---|---|---|---|
| 💡 | **画像から盤面取り込み (基本)** | L | カメラ撮影 / フォトライブラリ / ドラッグ&ドロップ / クリップボードから 1 枚の画像を受け取り、6×13 グリッドのセルごとに R / P / B / Y / おじゃま / 空 を判定して `field` に流し込む。判定後は通常の編集モードに入って手直しできるフローが理想 |
| 💡 | **NEXT / NEXT2 も合わせて取り込む** | M | 上記が動いたら次のステップ。スクショ全体から右側の NEXT エリアも切り出して 2 ペアぶん認識 |
| 💡 | **「現在のペア」の取り込み** | M | 落下中のペアは画像内の場所が一意でないので難度高め。ROI 検出 + 色分類 |

### 想定する実装アプローチ(難度順)

1. **手動クロップ + 色平均判定 (S+)**: ユーザーが「盤面の四隅」をタップしてマスクを与える → その矩形を 6×13 で等分 → 各セルの中心ピクセル色を取得 → 既知の puyo 色 (`PUYO_COLORS`) との RGB ユークリッド距離で最近傍判定。**MVP として真っ先に試したい**
2. **テンプレートマッチング (M)**: 各色の puyo を小テンプレ画像にして、`canvas` にロードした入力画像とスライドウィンドウで最大相関を取る。背景の縞・グリッド線に強い
3. **軽量 CNN 分類器 (L)**: Tensorflow.js で 32×32 の puyo パッチを 6 クラス分類(R/P/B/Y/G/empty)。学習データは `Board.tsx` のレンダラを使って自前で大量生成可能(人工データだけで十分精度出るはず)。プロジェクトに既に TensorFlow.js が入っているので増分コストが低い

(1) で 90% 当たるはずなので、まずは (1) で出して使い心地を見て、ハマる場面が明確になったら (2) / (3) に上げるのが現実的。

### UX 検討事項

- **画像のソース**:
  - `<input type="file" accept="image/*" capture="environment">` で iOS / Android のカメラ起動 + ライブラリ選択が両方賄える
  - クリップボード貼り付け: `navigator.clipboard.read()`(モバイルは制限あり)
  - URL 入力 / D&D は PC ユーザ向け
- **誤認識の救済**:
  - 取り込み完了 → 自動で **編集モードに遷移** → ユーザーが間違ったセルだけ手で直して Apply、というフロー
  - 認識結果の「自信度」が低いセルだけハイライト表示できると親切
- **対応ソース**:
  - **アプリ内のスクショ** (本シミュレータ自身): 一番楽。色固定・グリッド固定なので (1) でほぼ 100% 取れる
  - **ぷよぷよ eスポーツ / 通の公式画面**: 撮影条件と色の幅が広いので (3) が必要そう
  - **手描きのメモ** (色鉛筆など): カラーパレットを学習しないと厳しい。優先度低
- **プライバシー**: 画像は **すべてクライアント側で処理**、外部送信しないのを明記。サーバー保存タスクと同じ温度感

## ROI 観点で次に着手するならの推奨ショートリスト

過去のブレストの締めくくりで挙げた 3 つを引き続き推す:

1. **マッチリプレイ再生 + 共有 URL** — 既に保存済みデータを生かせて、教材性・拡散性ともに高い
2. **デイリーシード + 統計画面** — 毎日触る理由を作りつつ、過去のレコードに価値を生やす
3. **おじゃまサバイバル / タワー** — 既存の機能(おじゃま対応 / preset 切替)をフル活用してゲームの幅を広げる
